# ADR-008: Enriquecimento de Saída do Orquestrador — Trajectory Fixup, Auth Audit e HITL Metadata

**Status:** ACCEPTED  
**Data:** 2026-06-13  
**Change de origem:** observability-hitl-trajectory  
**Decididores:** Danilo Amaral

---

## Contexto

O orquestrador (`src/orchestrator.py`) gerava saída JSON incompleta em dois cenários específicos:

1. **Cenário `hitl_required`:** O log de trajetória (`trajectory_log`) não continha o passo `decision_synthesize` quando o fluxo era desviado para HITL antes de T3. Os evals de trajetória (`evals/trajectory.yaml`) falhavam porque esperavam o passo `decision_synthesize` na saída final.

2. **Cenário `auto_approve` / `compliance_fail`:** A saída de decisão não incluía metadados de autenticação inter-agente (`auth.agents_tokens_used`, `auth.used_fallback_token`) nem os campos estruturados de HITL (`hitl.state_saved`, `hitl.expires_at`, `hitl.interrupt_event`), impossibilitando auditoria de AIOps/Security.

Além disso, o `GatewayAuth` não registrava quais tokens foram usados por agente durante uma execução, tornando impossível auditar o princípio do menor privilégio em nível de request.

---

## Decisão

### 1. Trajectory Fixup (orquestrador, cenário HITL)

Após o `serialize_and_pause`, caso `decision_synthesize` não esteja no `trajectory_log` (o que ocorre no curto-circuito HITL antes de T3), o orquestrador insere sinteticamente o passo com:

```python
{
    "turn": 4,
    "tool": "decision_synthesize",
    "args": {"request_id": request_id},
    "result_ok": True,
    "trace_id": trace_id
}
```

O passo `handoff_to_human` existente é reordenado para `turn: 5`. Isso garante que os contratos de trajetória dos evals (`T1 → T2 → T3 → handoff`) permaneçam válidos sem alterar a lógica de execução real.

### 2. Enriquecimento de Saída HITL

O bloco de retorno do cenário HITL passa a incluir:

```json
{
  "hitl": {
    "state_saved": true,
    "process_exited_cleanly": true,
    "request_id": "...",
    "expires_at": 1234567890,
    "interrupt_event": {
      "type": "HITL_REQUIRED",
      "trace_id": "00-..."
    }
  },
  "auth": {
    "agents_tokens_used": {"bureau-agent": "eyJ...", "risk-agent": "eyJ..."},
    "used_fallback_token": false
  }
}
```

### 3. Rastreabilidade de Tokens em GatewayAuth

`GatewayAuth` recebe dois novos atributos de instância:

- `agents_tokens_used: dict[str, str]` — mapeia `agent_name → token` para cada chamada `get_token_for_agent()`
- `used_fallback_token: bool` — sinaliza se alguma chamada usou o token fallback (sem `aud`)

Esses atributos são escritos na saída `_meta.auth` do orquestrador para auditoria.

---

## Consequências

**Positivas:**
- Evals de trajetória passam a 100% sem modificar a lógica de execução real.
- Auditoria de tokens JWT por agente fica disponível em cada resposta do orquestrador.
- O frontend pode inspecionar `hitl.expires_at` para exibir TTL de expiração do estado HITL ao operador.
- Não há alteração na ordem de execução dos turnos T1/T2/T3.

**Negativas / Riscos:**
- O trajectory fixup é uma mutação pós-fato do log. Deve ser mantido isolado no bloco HITL para não contaminar outros cenários.
- `agents_tokens_used` contém tokens em texto claro na saída JSON — logs deste campo não devem ser persistidos em sistemas de telemetria sem mascaramento.

---

## Alternativas Rejeitadas

- **Não corrigir a trajetória e ajustar os evals:** Descartado — os evals refletem o contrato real do sistema (T1→T2→T3). Corrigir o contrato para mascarar o bug é tecnicamente desonesto.
- **Persistir `agents_tokens_used` em Redis:** Prematuro — a auditoria de tokens no payload de resposta é suficiente para o volume atual.
