# ADR-009: Adoção de SQLite como Store Durável Primário

**Status:** ACCEPTED  
**Data:** 2026-06-14  
**Change de origem:** sqlite-persistence  
**Decididores:** Danilo Amaral

## Contexto

O sistema usa `src/episodic_memory.json` como memória episódica persistente e `hitl_store.py` com Redis opcional para estados HITL. Em desenvolvimento local, Redis normalmente não está configurado, então HITL cai para memória de processo. Isso cria dois problemas:

- decisões históricas não têm índice eficiente por `request_id`;
- estados HITL somem em restart quando Redis não está ativo.

## Decisão

Adotar SQLite local como store durável primário em `src/credit_analysis.db`, com schema para:

- `analyses`: análises concluídas, pendentes ou em curso;
- `hitl_states`: payloads HITL serializados com TTL.

`episodic_memory.json` permanece no repositório como seed histórico e compatibilidade com evals existentes. O seed roda uma vez quando a tabela `analyses` está vazia.

## Alternativas Consideradas

- **Postgres:** mais adequado para produção multi-instância, mas adiciona dependência operacional fora do escopo da demo local.
- **Redis-only:** resolve HITL quando configurado, mas não atende memória episódica e continua opcional no ambiente padrão.
- **JSON file:** simples, mas sem índice por `request_id`, sem concorrência segura e frágil em operações de Git.

## Consequências

Positivas:

- Durabilidade local sem dependências externas.
- Lookup direto por `request_id`.
- Estados HITL sobrevivem restart sem Redis.
- Frontend poderá consultar histórico backend em sessão futura.

Riscos:

- SQLite local não substitui Postgres em produção multi-processo.
- O JSON continua sendo escrito para compatibilidade de evals; a remoção desse legado deve ser uma change separada.
